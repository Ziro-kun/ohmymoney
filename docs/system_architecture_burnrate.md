# Ohmymoney 자산 및 번레이트(Burn Rate) 시스템 구조 정의

이 문서는 Ohmymoney 서비스의 핵심인 실시간 자산 변동 및 고정 지출(Burn Rate) 산출 로직에 대한 데이터 구조와 연산 아키텍처를 정리한 문서입니다.

---

## 1. 데이터 스키마 정의 (SQLite)

### 1.1 Assets (자산/부채)
사용자의 기초 자산과 대출 정보를 관리합니다.
- `id`: 고유 식별자
- `name`: 자산/부채 명칭
- `amount`: 기초 잔액 (Seed Amount) - 과거의 기준점
- `type`: `asset` (자산) | `liability` (부채)
- `assetCategory`: `cash` | `stock` | `deposit` | `vehicle` | `loan` 등
- `depreciationRate`: 
    - 자산(`vehicle`)인 경우: **연간 감가상각률 (%)**
    - 부채(`loan`)인 경우: **연간 이자율 (%)** -> Cascade 엔진 연동 핵심 데이터

### 1.2 Transactions (거래 내역)
실제 발생한 현금 흐름과 정기 예약을 관리합니다.
- `description`: 내역 요약
- `amount`: 거래 금액
- `type`: `income` | `expense` | `transfer`
- `date`: 발생 일자 (YYYY-MM-DD)
- `isFixed`: 고정 지출 여부 (Boolean)
- `recurringDay`: 매월 정기 발생 일자 (1~31)
- `isVirtual`: 엔진에 의해 자동 생성된 가상 내역 여부 (Boolean)

---

## 2. 실시간 번레이트(Burn Rate) 산출 로직 (이벤트 주도형)

번레이트는 사용자의 순자산이 실시간으로 얼마나 '녹아내리고 있는지'를 나타내며, 네 가지 레이어의 합산으로 결정됩니다. **모든 계산은 미세한 틱 연산 과정에서의 부동소수점 오차 방지를 위해 `1/1000원 단위(Scaled Integer)`로 스케일링되어 계산된 후 실수형으로 반환**됩니다.

### 2.1 고정 거래 번레이트 (Fixed Transaction Burn)
`isFixed`가 활성화된 거래 중 **가상 내역이 아닌(Original)** 내역들을 합산하여 월 30일 기준으로 일할 계산합니다.
- **수식:** `Σ(고정 지출/이체 금액) / 30`

### 2.2 대출 이자 Cascade 번레이트 (Liability Interest Burn) - 실시간 잔액 기반(Dynamic)
부채(`liability`) 중 분류가 `loan`이고 `depreciationRate`(이자율)가 설정된 항목에 대해 **'현재까지의 모든 거래가 반영된 실시간 원금(Running Balance)'**을 기준으로 이자 비용을 산출합니다.
- **수식:** `Σ(대출의 실시간 원금 * 연이율 / 100) / 365`
- **특징:** 과거처럼 초기 대출 원금(`Seed`)에 의존하지 않습니다. 사용자가 대출 원금을 일부 상환(거래 내역 추가)하는 즉시, 이벤트 주도형 아키텍처에 의해 메모리 내의 실시간 잔액이 감소하고 이로 인해 **다음 틱(Tick)부터 번레이트 속도가 실시간으로 느려지는 것**을 체감할 수 있습니다.

### 2.3 자산 감가상각 번레이트 (Asset Depreciation Burn)
차량 등 가치가 하락하는 자산의 실시간 잔액에 대해 감가상각 비용을 산출합니다.
- **수식:** `Σ(차량의 실시간 가액 * 감가상각률 / 100) / 365`

### 2.4 최근 변동 지출 번레이트 (Recent Variable Burn) - 임시 번레이트 가중치
고정 지출만으로는 자산이 줄어드는 체감을 얻기 힘든 점을 보완하기 위해, 최근 7일 동안 발생한 **비고정 지출(Variable Expenses)**의 평균을 추가로 합산합니다.
- **수식:** `Σ(최근 7일간의 비고정 지출 금액) / 7`
- **특징:** 어제 과소비를 했다면, 오늘 번레이트 틱 속도가 빨라져 즉각적인 경각심을 줍니다.

## 3. 실시간 순자산(Net Worth) 반영 및 연산 프로세스

1. **기초 자산 로드:** DB에서 `assets` 테이블의 `amount`(Seed)를 가져옵니다.
2. **거래 내역 소급 적용:** 현재 날짜까지 발생한 모든 `transactions`(가상 내역 포함)을 순차적으로 가감하여 **현재 시점의 실제 잔액(Running Balance)**을 계산합니다.
3. **가상 내역 생성 엔진:**
    - `isFixed`이고 `recurringDay`가 설정된 내역에 대해, 과거부터 경과한 달의 가상 트랜잭션을 메모리 상에서 생성합니다.
4. **번레이트 재계산 트리거 (Event-driven):**
    - 거래 내역 테이블에 삽입, 수정, 삭제(CUD)가 발생할 때마다 별도의 DB 부하 없이 `useFinanceStore`의 메모리 상태(State)에서 즉시 자산의 실시간 잔액과 번레이트(`perSecondBurnRate`)를 재계산합니다.
5. **UI 바인딩 및 정밀도 처리:** 
    - 최종 산출된 `perSecondBurnRate`는 높은 소수점 정밀도를 유지하며 대시보드의 `netWorth`에서 매 시간 단위(ms)로 차감됩니다.
    - 단, 사용자 UI(대시보드 등)에 노출될 때는 **원 단위로 절사(0자리 소수점 표시)**하여 시각적 혼란을 방지합니다. (`src/utils/format.ts`의 기본 정밀도를 `0`으로 전역 설정)

---

## 4. 향후 확장 계획

- **복리 효과 반영:** AI 제안 카드에서 번레이트 지속 시 30일/1년 뒤 손실액을 복리로 계산하여 경각심 고취.
- **대출 상환 시뮬레이션:** 대출 원금 상환 시 줄어드는 이자 비용(Burn Rate 감소분)을 미리 계산해 주는 마법사 UI 도입.
